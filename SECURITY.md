# Política de Segurança do RavenFlow
### Reportar uma Vulnerabilidade
A segurança é uma prioridade para o RavenFlow. Se descobrir uma vulnerabilidade de segurança, agradecemos que nos informe de forma responsável.
Envie um email detalhado para **corporation.ravenflow@gmail.com** para reportar vulnerabilidades no RavenFlow. Mesmo que não tenha certeza se um bug é uma vulnerabilidade explorável, recomendamos enviar o reporte para **corporation.ravenflow@gmail.com** e não discutir o problema em outro lugar.
Discuta vulnerabilidades apenas por email, e não em público, até um anúncio oficial na lista de discussão do RavenFlow na data de lançamento.

Exemplos de detalhes para incluir:
- Idealmente uma breve descrição (ou um script) para demonstrar uma exploração
- As plataformas e cenários afetados (a vulnerabilidade pode afetar apenas configurações específicas, por exemplo)
- O nome e a afiliação dos pesquisadores de segurança que estão envolvidos na descoberta, se houver
- Se a vulnerabilidade já foi divulgada
- Quanto tempo um embargo seria necessário para ser seguro
Receberá uma confirmação do recebimento do reporte em até 48 horas, uma resposta inicial com avaliação da vulnerabilidade em até 7 dias, atualizações sobre o progresso da correção conforme necessário e reconhecimento público (se desejar) após a correção ser implementada.

## Versões Suportadas
Não há versões de "Suporte a longo prazo" oficiais no RavenFlow. As correções são feitas para a branch de manutenção da versão de feature mais recente (também conhecida como versão ".0") e recebem atualizações ocasionais com correções de bugs.
As correções para vulnerabilidades são feitas para a branch de manutenção da versão de feature mais recente e mescladas nas branches de desenvolvimento. O projeto RavenFlow não oferece garantia formal de que branches de manutenção mais antigas receberão atualizações, mas na prática, correções críticas de vulnerabilidades são aplicadas não apenas à branch mais recente, mas a pelo menos uma ou duas tracks de manutenção mais antigas.
Isso é normalmente feito aplicando a correção à branch de manutenção mais antiga ainda relevante e mesclando-a para cima para tracks de manutenção mais novas e mais recentes.
